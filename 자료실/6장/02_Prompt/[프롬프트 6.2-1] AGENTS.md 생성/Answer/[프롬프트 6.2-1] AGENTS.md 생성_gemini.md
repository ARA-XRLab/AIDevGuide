# AGENTS.md

이 문서는 SpaceShooter 프로젝트의 구조와 AI 에이전트의 역할 및 가이드라인을 설명합니다.

## 1. 프로젝트 요약
- **엔진 버전**: Unreal Engine 5.7
- **장르**: 2.5D 종스크롤 슈팅 게임
- **모듈 구조**: `SpaceShooter` 단일 모듈
- **핵심 플레이**: 플레이어는 총알을 발사하여 적을 파괴하고 점수를 획득하며, 적의 공격을 피해 살아남는 것이 목표입니다.

## 2. 주요 경로
- **C++ 소스 코드**: `Source/SpaceShooter/`
  - `Public/`: 헤더 파일 (.h)
  - `Private/`: 구현 파일 (.cpp)
- **에셋 위치**: `Content/Shooting/`
  - `Blueprints/`: 액터 및 시스템 블루프린트
  - `Materials/`: 머티리얼 및 인스턴스
  - `Models/`: 스태틱 메시 (Player, Enemy)
  - `Particles/`: VFX (Explosion 등)
  - `Sound/`: 사악 (사운드 큐)
- **프로젝트 파일**: `SpaceShooter.uproject`, `SpaceShooter.sln`

## 3. 런타임 흐름
1. **AShootingGameMode**: 게임의 중앙 제어 장치. 점수 관리, UI(MainWidget, GameOverWidget) 생성 및 업데이트를 담당합니다.
2. **APlayerPawn**: 플레이어 캐릭터. `EnhancedInput`을 통해 이동 및 사격(`FireBullet`)을 처리하며, 총알 풀링(`BulletPool`) 시스템을 내장하고 있습니다.
3. **AEnemyManagerActor**: 주기적으로 적(`AEnemyActor`)을 스폰 포인트 중 무작위 위치에서 생성합니다.
4. **AEnemyActor**: 아래로 이동하며 플레이어가 사정거리(`EncounterRadius`) 안에 들어오면 점사(`EnemyBulletClass`)를 수행합니다.
5. **Collision**:
   - `ABulletActor` (플레이어 총알) -> `AEnemyActor` 오버랩 시 데미지 및 파괴.
   - `AEnemyBulletActor` (적 총알) -> `APlayerPawn` 오버랩 시 데미지 처리.

## 4. 프로젝트 규칙
- **네이밍 규칙**: 
  - 클래스: `A`(Actor), `U`(Object/Widget), `I`(Interface) 접두사 사용.
  - 변수: PascalCase (예: `MaxHP`, `FireInterval`).
  - 함수: PascalCase (예: `OnDamage`, `FireBullet`).
- **에셋 로딩**: 생성자에서 `ConstructorHelpers::FObjectFinder`를 사용하여 하드코딩된 경로(`/Game/Shooting/...`)로 기본 에셋을 로드합니다.
- **충돌 프로필**: 주로 `BoxComponent`를 사용하며 `Overlap` 이벤트를 통해 충돌을 판정합니다.

## 5. 자주 하는 작업
- **입력 추가**: `IA_Move`, `IA_Fire` 외의 새로운 행동은 `Content/Shooting/Input/`의 `IMC_Player`에 매핑을 추가하고 `PlayerPawn.h/cpp`에서 바인딩합니다.
- **적 타입 추가**: `AEnemyActor`를 상속받는 새로운 클래스를 생성하거나, `EnemyManagerActor`의 `EnemyFactory`를 변경합니다.
- **UI 요소 추가**: `MainWidget` 또는 `GameOverWidget` (UMG)에 요소를 추가하고 `ShootingGameMode`를 통해 데이터를 전달합니다.

## 6. AI 에이전트의 책임과 한계
### 할 수 있는 것
- **코드 분석**: 프로젝트 구조를 파악하고 클래스 간의 관계를 분석합니다.
- **이슈 탐지**: 잠재적인 버그, 메모리 누수, 잘못된 엔진 API 사용 등을 식별합니다.
- **수정 제안**: 발견된 문제에 대한 C++ 코드 수정안을 제시합니다.
- **패턴 식별**: 프로젝트의 코딩 컨벤션을 학습하여 일관된 스타일의 코드를 작성합니다.

### 할 수 없는 것
- **최종 의사결정**: 아키텍처 변경이나 핵심 로직 수정에 대한 최종 승인은 개발자의 몫입니다.
- **프로덕션 배포**: 코드를 직접 빌드하거나 배포 환경에 적용하지 않습니다.
- **게임 밸런스 판단**: 수치(속도, 데미지 등) 변경이 게임의 재미나 난이도에 미치는 영향은 주관적이므로 판단하지 않습니다.

### 개발자 확인 필수
- **모든 코드 수정**: AI가 제안한 코드는 반드시 개발자가 검토 후 적용해야 합니다.
- **성능 영향**: 특히 `Tick` 함수 내부의 로직 변경이나 빈번한 가비지 컬렉션 유발 코드는 주의 깊게 확인해야 합니다.
- **게임플레이 변경**: 플레이 흐름이 바뀌는 작업은 기획 의도와 일치하는지 확인이 필요합니다.
